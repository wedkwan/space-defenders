# Plano: Unificar Lógica Solo/Multiplayer + WebRTC DataChannels Binário

## Objetivo

Unificar a lógica de jogo solo e multiplayer para que **o servidor NestJS seja sempre o autoritativo**, e substituir a comunicação Socket.io (JSON) por **WebRTC DataChannels com protocolo binário (ArrayBuffer)** para reduzir latência e tamanho de payload.

## Arquitetura Atual vs Alvo

**Atual:**
- Solo → lógica 100% client-side (browser), sem servidor
- Multiplayer → server-authoritative via Socket.io (JSON payloads)

**Alvo:**
- Solo → server-authoritative via WebRTC DataChannel (binary)
- Multiplayer → server-authoritative via WebRTC DataChannel (binary)
- Socket.io → mantido apenas para signaling (SDP/ICE exchange)

```
Browser
    │
    ├── Socket.io (signaling: SDP offer/answer + ICE candidates)
    │
    └── WebRTC DataChannel "game-data" (game payloads binários)
        ├── Client → Server:  inputs (ArrayBuffer, 3 bytes)
        └── Server → Client:  snapshots (ArrayBuffer, ~500-900 bytes)
```

### 2. Novo: `apps/game-server/src/game/binary-protocol.ts`

Protocolo binário compacto usando `ArrayBuffer` + `DataView` (little-endian):

**Input (Client → Server): 3 bytes fixos**
```
Offset 0: uint8  → tipo (0=move, 1=shoot)
Offset 1: int8   → direction (-1 ou 1, só para move)
Offset 2: uint8  → playerIndex (0 ou 1)
```

**Snapshot (Server → Client): header + arrays**
```
Header (4 bytes):
  uint8  wave
  uint8  status (0=waiting, 1=playing, 2=gameover)
  uint16 enemyCount

Players (11 bytes cada):
  uint8  id
  float32 x
  float32 y
  uint8  lives
  uint8  score (limitado a 255 para simplificar, ou usar uint16)
  uint16 invincible
  uint8  frame

Enemies (9 bytes cada):
  uint16 id
  float32 x
  float32 y
  uint8  width
  uint8  height
  uint8  frame
  uint8  group (0=left, 1=right)

Bullets (12 bytes cada):
  float32 x
  float32 y
  float32 vx
  float32 vy
  uint8  isPlayerBullet
  uint8  ownerId
```

Exportar funções:
- `encodeInput(type, direction, playerIndex): ArrayBuffer`
- `decodeInput(buffer: ArrayBuffer): { type, direction, playerIndex }`
- `encodeSnapshot(snapshot): ArrayBuffer`
- `decodeSnapshot(buffer: ArrayBuffer): GameSnapshot`

### 3. `apps/game-server/src/game/game.gateway.ts` — Refatorar

**Manter Socket.io apenas para signaling.** Adicionar:

- `@SubscribeMessage('webrtc:offer')` — recebe SDP do cliente, cria `RTCPeerConnection` (via `wrtc`), setRemoteDescription, cria DataChannel `game-data`, gera answer, envia de volta
- `@SubscribeMessage('webrtc:ice-candidate')` — repassar ICE candidates entre cliente e servidor
- DataChannel `onmessage` — decodifica input binário e chama `room.handleBinaryInput()`
- Quando `GameRoom` atualiza estado: chama `broadcastBinarySnapshot()` que envia via DataChannel

Estrutura de dados por conexão:
```typescript
interface PeerConnection {

  peerConnection: RTCPeerConnection; // via wrtc
  dataChannel: RTCDataChannel;
  roomId: string;
  playerId: string;
}
```

```typescript

  // Conectar: Socket.io → WebRTC signaling → DataChannel
  useEffect(() => { ... }, []);

  return { snapshot, connected, error, sendInput };

```

Fluxo interno:
1. Conecta Socket.io (signaling)
2. Emite `game:join` → recebe `game:joined` com roomId
3. Cria `RTCPeerConnection` com STUN server (`stun:stun.l.google.com:19302`)
4. Cria DataChannel `game-data` com `{ ordered: false, maxRetransmits: 0 }` (UDP-like)
5. Gera SDP offer → envia via Socket.io `webrtc:offer`
6. Recebe answer → setRemoteDescription
7. Troca ICE candidates via Socket.io
8. DataChannel abre → a partir daqui, tudo binário
9. `dataChannel.onmessage` → decoda snapshot → atualiza state

## Fluxo de Conexão Completo

```
1. Usuário clica "PLAY SOLO" ou "MULTIPLAYER"
2. Frontend conecta Socket.io ao game-server:4000
3. Emite: game:join { playerId, name }
4. Server cria sala (ou junta a existente), retorna: game:joined { roomId }
5. Server cria RTCPeerConnection (wrtc), cria DataChannel "game-data"
6. Server gera SDP offer → envia: webrtc:offer { sdp: offer.sdp }
7. Frontend recebe offer, cria RTCPeerConnection (browser nativo)
8. Frontend setRemoteDescription(offer)
9. Frontend cria DataChannel "game-data"
10. Frontend gera answer → envia: webrtc:answer { sdp: answer.sdp }
11. ICE candidates trocados via: webrtc:ice-candidate { candidate }
12. WebRTC connection conecta! DataChannel "game-data" abre
13. A partir daqui: todo game data vai pelo DataChannel em binário
14. Socket.io pode ser desconectado (ou mantido como fallback)
15. Game loop roda a 60Hz no servidor, envia snapshots binários
16. Frontend recebe snapshots, decodifica, renderiza no canvas
```

## Riscos e Mitigações

| Risco | Mitigação |
|-------|-----------|
| `wrtc` pode ter problemas de instalação (precisa de gcc, python3) | Documentar pré-requisitos; fallback para Socket.io + binary se necessário |
| `wrtc` pode não suportar todos os recursos do WebRTC | Usar apenas DataChannel (não media streams), que é bem suportado |
| SDP serialization entre `wrtc` (Node) e browser | Usar `.toJSON()` / `new RTCSessionDescription(json)` que é padrão |
| WebRTC NAT traversal em ambientes restritos | Usar STUN público do Google; adicionar TURN se necessário |
| Complexidade aumenta com 2 transports (Socket.io + WebRTC) | Socket.io fica apenas para signaling, bem delimitado |

## Notas Técnicas

- Binary protocol: **little-endian** (padrão x86/x64)
- DataChannel config: `{ ordered: false, maxRetransmits: 0 }` → UDP-like, mínima latência
- STUN server (dev): `stun:stun.l.google.com:19302`
- Player IDs no snapshot: mapear para uint8 (0 ou 1) para economizar espaço
- Score no snapshot: usar uint16 (max 65535) — suficiente para o jogo
- Se `wrtc` falhar: fallback documentado para Socket.io com `binaryType: 'arraybuffer'`
